# 智慧物业 OA · 工单协同管理系统

## 项目概述

物业工单协同管理系统，前端纯 HTML/CSS/JS，后端 Node.js + Express，数据存储在飞书多维表格中。网页上所有操作（派单、驳回、完成）均会实时写回飞书。

---

## 系统运行原理

```
┌─────────────┐         ┌──────────────────┐         ┌─────────────────┐
│  浏览器页面  │ ──────→ │  Node.js 后端     │ ──────→ │  飞书多维表格    │
│  (前端)      │ ←────── │  (Express 服务器) │ ←────── │  (数据存储)      │
└─────────────┘  HTTP   └──────────────────┘  飞书API └─────────────────┘
     :3001                    :3001
```

**数据流**：
1. 浏览器打开 `http://localhost:3001`
2. 前端发起 `GET /api/tickets`
3. 后端用 App ID + App Secret 获取飞书 `tenant_access_token`
4. 后端调用飞书多维表格 API 读取全部记录
5. 后端将飞书字段（中文）转换为前端字段（英文代码），返回 JSON
6. 前端渲染工单列表、看板、日程等

**操作写回**：
- 派单 → `PATCH /api/tickets/:id` → 后端更新飞书"负责人"+"工单状态"字段
- 驳回 → `PATCH /api/tickets/:id` → 后端更新飞书"工单状态"+"驳回原因"字段
- 确认完成 → `PATCH /api/tickets/:id` → 后端更新飞书"工单状态"+"完成时间"字段
- 新建工单 → `POST /api/tickets` → 后端在飞书新增一条记录

**容错**：如果后端 API 不可用，前端自动回退到 localStorage 本地数据。

## 本地运行

```bash
# 1. 安装依赖
cd server
npm install

# 2. 启动服务
node index.js

# 3. 浏览器打开
http://localhost:3001
```

服务器同时托管前端静态文件和后端 API，所以只需一个端口。

## 功能模块

### 1. 运营看板（Dashboard）
- KPI 卡片：总工单量、报修、投诉、帮助/其他、紧急待处理、平均时长、按时完成率
- ECharts 图表：近 30 天工单趋势、各师傅处理量、平均处理时长、类型分布、状态分布、事件频率
- 师傅/管家绩效表：处理类型、完成量、处理中、平均时长、按时率、综合评分

### 2. 报修工单
- 五类：水暖 / 电路 / 电器 / 门窗 / 公共设施
- 流程：发起 → 派单 → 现场确认 → 维修 → 结果确认 → 完成
- 四级优先级（紧急/高/普通/低）颜色标签
- 按优先级 + 等待时间排序

### 3. 投诉工单
- 四类：突发事件 / 物业服务 / 便民服务 / 其他
- 流程：发起 → 指派 → 处理 → 结果确认 → 完成

### 4. 帮助/其他工单
- 四类：生活帮助 / 咨询建议 / 邻里协调 / 其他
- 流程：物业主管指派 → 管家处理 → 主管确认/驳回

### 5. 师傅日程（时间轴）
- 按小时刻度展示（7:00-22:00）
- 工单占位宽度 = 该师傅历史平均维修时长
- 无历史数据时按类别默认值（水暖 2.5h、电路 2h 等）
- 同师傅同时段重叠 → 红框高亮 + 底部冲突提示
- 日/周/月视图切换
- 按人员筛选

### 6. 管理平台
- 维修工/物业管家增删改查
- 数据存储在 localStorage（不依赖飞书）

## 角色与权限

| 角色 | 权限 |
|------|------|
| 工程部主管 | 报修工单派单、确认完成、驳回 |
| 物业主管 | 投诉/帮助工单指派、确认完成、驳回 |
| 维修工（张/李/王/赵） | 上传照片、提交处理结果（仅自己负责的报修） |
| 物业管家 | 提交投诉/帮助工单处理结果 |

**关键规则**：
- 师傅之间禁止互相转单
- 仅主管可驳回工单，驳回必须填写原因
- 驳回后工单退回原负责人重新处理

## 工单优先级

| 级别 | 颜色 | SLA 阈值 | 触发关键词 |
|------|------|----------|-----------|
| 紧急 | 红色 | 2 小时 | 爆裂、燃气、消防、断电 |
| 高 | 橙色 | 8 小时 | 漏水、占用、损坏、噪音 |
| 普通 | 蓝色 | 24 小时 | 默认 |
| 低 | 灰色 | 48 小时 | 建议、咨询 |

## API 接口

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/tickets` | 获取全部工单（从飞书读取） |
| POST | `/api/tickets` | 创建新工单（写入飞书） |
| PATCH | `/api/tickets/:recordId` | 更新工单（写回飞书） |

### PATCH 请求体示例

```json
{
  "status": "doing",
  "worker": "张师傅"
}
```

```json
{
  "status": "done",
  "finished": "2026-07-17T15:30:00.000Z"
}
```

```json
{
  "status": "doing",
  "rejectReason": "现场照片不完整"
}
```

## 技术栈

- **前端**：原生 HTML5 + CSS3 + JavaScript ES6 + ECharts 5.x
- **后端**：Node.js + Express + node-fetch
- **数据**：飞书多维表格（通过开放 API 读写）
- **人员管理**：浏览器 localStorage
- **部署目标**：Render（免费 Web Service）

## Render 部署指南

1. 把 `server/` 目录推送到 GitHub 仓库（只推 server 目录即可）
2. 在 [Render](https://render.com) 创建 **Web Service**
3. 连接你的 GitHub 仓库
4. 设置：
   - **Root Directory**: `server`（如果整个仓库就是 server 内容则留空）
   - **Build Command**: `npm install`
   - **Start Command**: `node index.js`
   - **Node Version**: 18 或以上
5. 在 Render 后台 → Environment → 添加环境变量：
   - `FEISHU_APP_ID` = 你的 App ID
   - `FEISHU_APP_SECRET` = 你的 App Secret
   - `FEISHU_APP_TOKEN` = 你的多维表格 app_token
   - `FEISHU_TABLE_ID` = 你的数据表 table_id
6. 点击 Deploy，等待构建完成
7. 访问 Render 分配的 URL（如 `https://property-oa.onrender.com`）

**注意**：
- `.env` 文件已在 `.gitignore` 中，不会上传到 GitHub
- Render 免费版服务器闲置 15 分钟会休眠，首次访问需等 30-60 秒唤醒
- 前端静态文件在 `server/public/` 中，后端自动托管

## 关于飞书表格编辑权限

当前多维表格由应用 API 创建，所有者是应用本身。个人账号查看/编辑需要申请 `drive:drive` 或 `drive:file` 权限后，通过 API 添加协作者。

**但这不影响系统正常运行**——所有数据操作都通过网页 → 后端 → 飞书 API 完成，不需要手动打开飞书表格。
